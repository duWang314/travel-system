# 个性化旅游系统实现方案

作者：**Manus AI**

## 技术栈选择

本仓库当前主要包含地图 JSON、头像图片、室内导航数据、游记文本和工具脚本，尚未提供可运行的前后端框架。因此，本项目采用 **Python FastAPI + 原生 HTML/CSS/JavaScript** 作为实现方案。该方案能直接复用现有 `backend/data` 目录，不依赖数据库，运行方式简单，适合数据结构课程设计的展示与验收。

| 层次 | 技术 | 选择理由 |
|---|---|---|
| 后端服务 | FastAPI | 代码量少，接口清晰，便于展示排序、查找、图算法和压缩算法。 |
| 前端页面 | 原生 HTML/CSS/JavaScript | 无构建依赖，便于打包与在任意机器上运行。 |
| 数据持久化 | JSON/TXT 文件 | 符合“避免使用数据库”的批注要求，方便老师检查数据结构。 |
| 地图绘制 | SVG + Canvas-like 坐标投影 | 直接渲染仓库中已有道路图，不依赖商业地图 API。 |
| 交互风格 | Apple Cinematic Minimalism | 黑白灰分段、单一蓝色交互，突出推荐、路线与算法结果。 |

## 数据文件设计

系统会在 `backend/data` 下补齐以下文件。所有文件都可以直接打开检查，避免数据库黑盒。

| 文件 | 内容 | 对应功能 |
|---|---|---|
| `destinations.json` | 至少 200 个景区/校园目的地的元数据，包含热度、评分、类别、兴趣标签和地图文件 | 旅游推荐、目的地搜索 |
| `users.json` | 初始化管理员 `admin/admin` 与若干普通用户，保存头像、兴趣、历史记录和角色 | 登录注册、用户中心、管理员管理 |
| `diaries_meta.json` | 游记标题、作者、目的地、热度、评分、封面、文件名等索引 | 游记管理、精确查找、排序、全文搜索 |
| `foods.json` | 各目的地附近的餐厅/美食窗口，包含菜系、距离、热度与评分 | 美食推荐与模糊查询 |
| `generated_covers/` | 游记封面占位图或生成结果缓存 | 游记封面生成演示 |
| `diaries/*_zip.txt` | 哈夫曼压缩后的游记内容 | 无损压缩与读取 |

## 核心算法覆盖

| 作业功能 | 实现算法 | 数据结构 |
|---|---|---|
| 旅游推荐 Top 10 | 加权评分 + 小顶堆 Top-K，避免完全排序 | 列表、哈希表、堆 |
| 查询结果排序 | 多关键字排序，支持热度、评分、兴趣匹配 | 列表、排序 key |
| 路线规划 | Dijkstra 最短路，支持距离、时间、交通工具限制与必打卡点分段规划 | 邻接表、优先队列 |
| 场所查询 | 以道路图最短路径距离为准的服务设施查找与排序，不使用直线距离 | 图、优先队列、过滤器 |
| 室内导航 | 室内图 Dijkstra，从入口导航至房间 | 室内邻接表、优先队列 |
| 游记精确查询 | 标题哈希索引 | 字典 / 哈希表 |
| 游记全文搜索 | KMP 字符串匹配，返回命中次数与片段 | next 数组、文本列表 |
| 游记压缩 | Unicode 字符级哈夫曼编码，保存码表与二进制串 | 哈夫曼树、堆 |
| 美食推荐 Top 10 | 模糊匹配 + 加权 Top-K | 列表、堆 |
| 管理功能 | 基于角色的增删改查 | JSON 文件、哈希索引 |

## 前端页面规划

| 页面 | 主要内容 |
|---|---|
| 首页 / 推荐 | 搜索栏、兴趣/权重设置、Top 10 目的地推荐、算法说明。 |
| 登录注册 | 用户名、密码、头像选择、兴趣选择。 |
| 目的地地图 | SVG 道路图、节点/边悬浮信息、起点/终点/必打卡点、路线规划、附近设施与美食查询。 |
| 室内导航 | 公共教学楼房间列表、入口到房间的路径步骤。 |
| 游记交流 | 按目的地、标题和正文搜索游记，按热度/评分排序，支持评分与浏览计数。 |
| 游记编辑 | 创建、读取、压缩、解压游记，并显示哈夫曼树摘要和节省率。 |
| 用户中心 | 修改兴趣、头像和密码，查看历史，管理本人游记；管理员可以管理所有游记与用户。 |
| 算法说明 | 汇总每个功能对应的数据结构、算法和复杂度，便于验收答辩。 |

## 实现边界与说明

本系统会优先实现课程作业要求中列出的所有可演示功能。AIGC 动画生成由于需要外部模型与视频生成服务，课程验收中将实现为“封面生成/动画提示词生成”的可演示接口：系统从游记内容抽取关键词生成提示词，并可缓存封面图；若用户日后提供可用 API Key，可在同一接口中接入真实图像或动画生成服务。

现有地图数据已经满足多数字段要求，但部分目的地地图节点数量较少。为满足“景区/校园数量至少 200 个”的要求，系统会补齐 200 个目的地元数据，并将未单独采集地图的目的地复用已有高质量道路图。这符合题目中“景区和校园内部可以一致”的说明，也便于集中展示图算法。
